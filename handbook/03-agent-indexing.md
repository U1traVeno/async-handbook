# Agent 如何使用 Handbook

Agent 参与异步协作，可以帮助人们找到背景、发现断裂并减少重复解释。它不负责替团队执行机械检查。Handbook 应当让 Agent 和人类读取同一份规范，团队无需另外维护字段化的隐藏规则。

## Agent 需要什么样的上下文

Agent 要可靠地提供帮助，至少需要知道当前协作对象、用户希望完成的事情，以及可能适用的规范。具体任务还可能需要项目背景、权限范围、历史决定或平台状态。这些信息不一定都写在当前页面中，但应能沿链接找到权威来源。

标题、自然语言、内部链接和版本历史通常比预先设计的标签体系更有用。只有当知识规模或工具能力已经证明需要结构化信息时，才增加元数据；不应凭想象要求每页提供 `type`、`owner`、`status`、`review_at` 或其他字段。

Agent 找不到信息时，应区分三种情况：信息确实没有记录、信息可能存在但当前不可访问、信息存在但位置或名称难以发现。只有第一种情况能够支持“缺少记录”的判断，后两种情况应说明限制并请求线索或权限。

## 查阅规范

Agent 应先根据用户的目标定位直接相关章节，再沿内部链接理解上位原则、术语、例外和平台落地。不得只匹配一个关键词或标题，就忽略规则成立的条件。

引用规范时，应指向具体页面和标题，并用自己的话说明这条规范为什么与当前情况有关。如果引用的是某个 Git 版本或历史决定，应说明版本或时间，避免把已经变化的内容当作当前规则。

Handbook 没有覆盖某个问题时，Agent 可以结合事实提出一般性建议，但必须把建议与 Handbook 要求区分开来。不得借“最佳实践”之名创造组织尚未采用的义务。

## 审阅协作产物

Agent 审阅时应判断这份内容能否让相关人员可靠地继续工作。字段只能提供部分证据。一次有效审阅通常包含：

- 所依据的规范和当前可见事实；
- 观察到的信息断裂、冲突或风险；
- 这一问题可能怎样影响后续协作；
- 与风险相称的改进建议；
- 依赖推断之处的不确定性。

结构信息可以成为证据，但不能替代判断。例如，没有单独的“Owner”字段不代表无人负责，正文和任务分配可能已经清楚说明负责人；存在“更新时间”也不证明内容仍然正确。

Agent 可以明确指出可直接观察的事实，例如链接失效或两个权威页面给出冲突结论。对于“背景是否充分”“截止时间是否合理”“方案是否正确”等依赖场景的判断，应使用风险提示或问题，避免直接宣布违规。

## 采取行动的边界

Agent 可以在用户授权范围内查找、整理、比较和修改协作内容，也可以提出下一步。它不得：

- 代替 DRI 作出需要业务取舍的决定；
- 将没有规范依据的偏好写成强制要求；
- 因无法访问受限内容就断言内容不存在；
- 根据措辞、响应速度或格式给个人评定善意、能力或文化分数；
- 为了提高自动检查率而要求团队增加无实际协作价值的字段；
- 在没有确认影响范围时批量修正文档或覆盖人的决定。

涉及敏感信息时，Agent 应遵守与人类参与者相同的权限边界。提高可检索性不意味着扩大可见范围；必要上下文可以存放在受控位置，并向有权参与的人和 Agent 开放。

## 自动化应从真实问题开始

自动化适合处理重复、边界清楚且误判成本低的问题，例如验证内部链接、提示明确的状态冲突或汇总已有记录。实际发生频率和处理成本决定自动化是否值得投入，技术可行性只是其中一个条件。

对过期页面、无负责人任务、会议无纪要等现象，不应预设统一扫描规则。团队应先确认相应信息在什么场景确实必要，再定义证据、例外、误报处理和人工纠正路径。

自动化的输出应可追溯、可纠正并允许人类覆盖。Agent 被纠正后，应更新当前判断或提出 Handbook 的缺口，不再坚持未被正文支持的规则。

## Handbook Skill 的职责

Handbook Skill 可以描述如何定位章节、沿链接补齐上下文、理解规范用语和引用版本。Skill 不复制正文，不为每类协作对象维护独立检查器，也不把建议变成隐藏要求。

当 Handbook 结构变化时，Skill 只需更新受影响的查阅路径。规范含义发生变化时，应先修改 Handbook，再让 Agent 使用新版本。
