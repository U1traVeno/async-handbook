# 异步协作介绍

本手册最终目的是希望更好地帮助团队落实异步工作制度。因此有必要在此介绍异步协作制度的基本概念、优势，以及落实异步协作的简单建议。

## 为什么需要异步协作

过去，团队协作的主要瓶颈往往是代码开发本身；而今天，越来越多团队发现，真正限制产出的开始变成了协作。

Agent 工具大幅降低了实现成本。每个人不仅能够更快地产生代码，也能够更快地产生设计方案、分析报告、文档、Review 意见和讨论内容。真正开始爆炸增长的，不只是代码量，更是每个人需要处理的信息量。

然而，大多数团队的协作方式并没有随之改变，仍然依赖聊天软件、临时会议和口头沟通作为主要协作方式。

这些方式并不会随着团队规模扩大而自然扩展。相反，它们会逐渐暴露出共同的问题：

- 重要需求散落在聊天记录中；
- 决策停留在会议里，没有留下统一记录；
- 项目进度依赖少数人的记忆才能了解；
- 新成员需要翻阅大量历史消息才能理解背景；
- 作息不同或无法同时在线的线上成员难以及时参与讨论；
- AI Agent 无法理解"昨天会里讨论的那个方案"这样的隐含上下文。

而继续增加聊天群、频道或会议，并不能解决这些问题。它们只是把更多信息分散到了更多地方。随着信息持续增长，真正的问题已经不是沟通得够不够多，而是信息是否拥有一个长期、稳定、可引用，可快速索引的位置。关于这一问题的完整讨论和真实案例，可以参考[Veno 的博客: 低效沟通的灾难与异步协作的探索](https://v3n0.top/post/paramer/2/)

异步协作希望解决的，正是这些问题。它并不意味着取消同步沟通，而是将重要信息从即时交流中抽离出来，沉淀到文档、Issue、Pull Request 等长期载体，使团队成员能够在不同时间理解上下文、推进工作，并使后来参与者以及 AI Agent 都能够依据同一份事实继续协作。

这样做能够带来许多直接收益：

- **聊天不会成为唯一的信息来源**。 重要背景、决策和结论会沉淀到可搜索、可链接、可引用的长期载体，而不是停留在聊天记录中。
- **会议不再承担信息同步的职责**。 同步沟通可以专注于讨论、决策和解决分歧，而不是重复介绍背景。
- **上下文能够持续积累**。 新成员、后来参与者，甚至几个月后的自己，都能够理解事情为什么这样做，而不是依赖他人的记忆。
- **不同作息和工作时间的人仍然能够协作**。 即使团队成员分布在不同城市、不同国家，或者只是拥有不同的生活节奏，也无需依赖所有人同时在线。
- **工作能够更加专注和高效**。 减少频繁的即时沟通和上下文切换后，成员能够拥有更长时间的连续专注（Deep Work），在完成思考后再进行沟通和反馈，从而提升工作质量和整体产出，而不是不断响应消息。
- **每个人拥有更大的自主权和行动空间**。异步协作鼓励团队成员在职责范围内自主推进工作，而不是默认等待许可。当背景信息、目标和决策边界都能够通过文档和长期记录获得时，大量工作无需依赖某个人在线确认才能继续进行。对于管理者而言，与其要求成员事事请示，不如建立清晰的职责边界，相信团队成员能够独立判断，并在遇到不确定的问题时提供咨询和支持。这不仅减少了等待和打断，也让团队能够以更高的效率持续推进工作。
- **沟通更加深思熟虑**。 相较于即时讨论，文档、Issue 和 Pull Request 等异步沟通方式通常给予参与者更多整理思路和查证信息的时间，使观点更加完整、决策过程更加透明，也减少因仓促表达带来的误解。
- **工作压力更加可控**。 当团队不再默认要求即时回复，每个人都能够根据工作的优先级安排回复和处理事务，减少持续在线和消息轰炸带来的心理压力，也有助于形成更加健康、可持续的工作节奏。
- **Agent 能够真正参与协作**。 AI 无法读会议室里的空气，也无法理解聊天中的隐含语境，但能够阅读文档、Issue、Pull Request 和 Handbook，并依据这些长期记录提供帮助。
- **信息增长不会压垮团队**。 当 AI 大幅提升每个人的产出速度后，团队真正需要管理的不再只是代码，而是越来越多的沟通、决策和知识。异步协作不是减少信息，而是让信息始终拥有清晰的归属和可追踪的位置。

后续章节中的所有协作规范——无论是文档、Issue、Pull Request、任务管理，还是 AI Agent 的参与方式——都围绕这一目标展开。

## 什么是异步协作

异步（asynchrony），概念与同步相对，指信息的发送方与接收方不需要共同的时钟。

在计算机系统中，异步通常指不阻塞当前线程来等待处理完成，而是允许后续操作继续执行，直至其他线程完成处理并回调通知当前线程。

异步在沟通场景中，指双方不在同一时间交换信息，典型的例子就是阅读 Issue/PR。相较于常见的微信、电话和线上会议等同步沟通，异步沟通能减少对心智的占用，给予双方更多思考的空间，也更方便记录沟通信息。

而异步协作是一种以文档等长期记录的形式为基础，让团队成员能够在不同时间独立推进工作的协作方式。它强调共享上下文，而不是共享在线时间。

基于异步沟通之上的异步协作，可以在协作中给对方预留出思考和处理的时间，不打扰其专注的工作状态，从而收获更高质量的输出，非常适合跨时区合作和远程办公。并且已经成为现代工作团队的一种趋势。

## 何时选择异步沟通，何时选择同步沟通？

虽然更倾向于采用异步沟通，但真正高效的团队需要在同步与异步之间取得平衡。

异步本身并不是目的。真正重要的是，在适合的时候选择异步沟通，把同步交流留给那些确实需要实时讨论的事情。

成熟的异步协作并不排斥同步会议，而是在恰当的时候主动使用同步沟通。

这里可以参考 [GitLab 团队成员如何选择同步与异步沟通](https://handbook.gitlab.com/handbook/company/culture/all-remote/asynchronous/#gitlab-experts-advise-on-when-to-use-sync-vs-async)：

> - 遇到事故、紧急事件或临近截止日期时，我会选择同步会议，帮助团队尽快解决问题。
> - 对于故障排查，实时讨论通常比来回发送消息更高效。
> - 当异步沟通已经无法继续推进事情时，我会切换到同步沟通。
> - 团队一起头脑风暴、讨论创意或方案时，同步交流通常更有效。
> - 与其花几个小时在 Slack 上回复上百条消息，或者等待几天才能收到回复，不如直接开一个十分钟的视频会议。
> - 工作相关、正式的沟通，我默认选择异步；而建立团队关系、社交聊天，则更适合同步进行。
> - 如果团队成员彼此还不熟悉，我会通过一次同步交流建立最初的人际联系，例如 Coffee Chat 或团队会议。
> - 如果事情对时效要求很高，我会召集所有相关人员同步讨论，以便尽快推进。
> - 我通常先尝试异步沟通；如果因为时效性、沟通偏差或讨论陷入僵局而无法继续推进，就会切换到同步讨论，待问题解决后再回归异步。
> - 异步沟通非常适合深入的技术讨论，尤其是代码相关的话题，因为很多内容需要反复阅读和思考。对于宏观问题，则可以结合异步沟通与视频会议，但最终结论应记录到 GitLab Issue 中，以保证透明和可追溯。
> - 作为产品设计师，我会在设计初期安排同步讨论，以充分理解问题并与团队共同头脑风暴；进入执行阶段后，则更多依赖异步沟通。
> - 我的大部分沟通都是异步进行的，但偶尔举行一次同步会议，有助于整个产品团队重新达成一致。
> - 我通常默认采用异步沟通，因为它更容易追溯决策过程。如果围绕同一个问题反复沟通多次，我会安排一次简短的同步讨论，把问题说清楚，然后继续回到异步协作。
> - 有些技术概念通过文字理解成本很高，这时直接与技术同事同步交流往往效率更高；理解之后，再继续异步协作即可。
> - 在等待回复的过程中，异步沟通常常需要频繁切换上下文，因此有时通过一次同步讨论快速解决问题，反而是效率最高的做法。

## 怎样进行异步沟通？

归根结底，异步沟通其实很简单。我们每天都在使用它，例如发送消息、留下语音留言、录制视频等。所谓异步沟通，就是消息的发送者和接收者通常不会在同一时间参与这次沟通。

不过，要真正做好异步沟通，需要投入更多思考。在发送一条异步消息之前，不妨先问问自己：

- 我是否选择了最合适的沟通形式（文字、语音、视频等）？
- 接收者是否已经具备理解这条消息所需的全部背景信息？
- 我的表达是否足够清晰，不会引起歧义或误解？
- 我是否考虑了措辞和语气，以及对方可能会如何理解这条消息？
- 我是否提供了必要的资料、链接或后续步骤，让讨论或项目能够继续推进？
- 这次沟通是否会被妥善记录，方便今后查阅和参考？

这样的思考，往往能让沟通更加清晰、完整、友善，也更容易推动事情取得进展。

有一个很好的切换到异步沟通思维方式的办法：问问自己，如果我的团队中其他人都在睡觉，我当下应当怎样高效的传递我的信息，进行我的工作，推动项目进展？

这也意味着，异步沟通需要投入更多时间进行准备和规划，并依赖合适的工具。要真正适应异步工作，不仅需要学习新的沟通方式，也需要主动改变过去的一些沟通习惯。

这些投入都是值得的。高质量的异步沟通能够显著提升团队效率，并促进更好的协作。
