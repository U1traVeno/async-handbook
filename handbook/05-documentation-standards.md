# 文档编写规范

## 页面定位

本页定义普通协作文档的写作规范。它不同于 [Handbook 编写规范](02-handbook-writing-guide.md)：后者规定 Handbook 本身如何写，本页规定组织日常文档、项目说明、方案说明和知识文章应如何写。

## 内容梗概

本页应提供一套简洁、可执行、可检索的文档写作方式，让读者能快速判断文档是否说明了背景、目标、适用范围、结论、下一步和维护责任。

## 建议大纲

### 文档何时值得创建

- 信息会被重复使用。
- 结论会影响后续行动。
- 多人需要基于同一背景协作。
- 口头或聊天传递会造成遗漏、误解或重复解释。

### 推荐结构

- 摘要：一句话说明文档回答什么问题。
- 背景：为什么需要这份文档。
- 目标：读者阅读后应能做出什么判断或行动。
- 范围：适用和不适用的场景。
- 正文：规则、方案、分析或步骤。
- 结论：当前采用的判断或下一步。
- 维护信息：Owner、状态、更新时间、复核时间。

### 写作风格

- 先结论，后解释。
- 标题具体，便于搜索。
- 段落短小，但不把完整判断切成口号。
- 首次出现的术语应解释。
- 避免把关键含义藏在截图、视频和附件中。

### 可检索性

- 标题应包含真实主题，而不是内部玩笑或临时代号。
- 关键术语应与术语表一致。
- 相关页面应显式互链。
- 过期内容应标记，不应继续被当作事实引用。

### 示例要求

- 重要规则或流程应提供正例。
- 容易误用的规则应提供反例。
- 示例应脱敏，避免泄露客户、凭据和个人信息。

### Agent 可检查性

- Agent 可以检查标题、Owner、状态、链接、更新时间和结构完整性。
- Agent 可以提示上下文不足、结论不明确或信息源冲突。
- Agent 不应在没有业务背景时判断方案是否正确。
