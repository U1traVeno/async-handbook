# Handbook 编写规范

本文定义本手册本身应如何编写、审阅和维护。最终希望确保每项规范有明确理由，并且可执行含义，不含糊，使人类与 AI Agent 能够阅读同一份内容并得到一致理解。

本文同时包含强制规范和写作建议。出现“必须”“应当”“可以”“不得”时，含义见下文[**“规范用语”章节**](#规范用语)；其他表述默认是解释或建议。

## 为什么需要编写规范

一本协作手册很容易出现三类问题：

- 内容持续增加，但团队无法判断哪些是有效规范、哪些只是建议。
- 同一个流程在多个页面重复出现，更新后产生互相矛盾的版本。
- 为了便于自动检查而堆积字段和术语，最终人不愿意阅读，AI 也无法理解规则背后的条件和边界。

因此，手册的质量不能以页面数量衡量。每一页都应解决一个真实、可重复出现的协作问题，并说明必要边界。可以先提交最小但有用的初稿，再通过小型 Pull/Merge Request 持续完善；只有问题尚不清楚或没有具体文本可讨论时，才先使用 Issue。

## 规范用语

- **必须**：满足规范所必需的要求。违反后会使协作产物不完整、不安全或不可追踪。
- **不得**：明确禁止的行为。
- **应当**：通常需要遵守，但在合理例外下可以偏离；偏离者应能说明原因。
- **可以**：可选做法，用于提供路径而不是增加义务。

不要在同一含义上混用“需要”“最好”“尽量”“建议”等模糊词。若一项要求存在例外，应写明例外条件，而不是用模糊语气掩盖边界。

## 收录标准

一项内容进入 Handbook 前，应同时满足以下条件：

1. 它处理的是团队会重复遇到的协作问题。
2. 不写清楚时，可能造成信息丢失、责任不明、等待、返工或决策不可追踪。
3. 它能够描述适用对象、触发条件、预期行为和合理例外。
4. 它与现有规范不重复；若有关联，可以链接到现有权威页面。

以下内容通常不应进入 Handbook：

- 一次性项目安排、临时通知或事件复盘。
- 只说明按钮位置的工具教程。
- 尚未被采用的愿景、口号或个人偏好。
- 薪资、法务、财务、行政制度和企业文化宣传；但与反馈、包容、信任和可持续协作直接相关的人文原则不在此限。
- 无法说明适用范围的宽泛“最佳实践”。

平台操作说明可以作为 Handbook 内的 Implementation 或 Example 存在，但不得冒充跨平台规范，也不得成为 Handbook 之外的第二套规范。

## 写作检查清单

Handbook 页面不需要套用固定模板。作者可以按问题本身选择自然的叙述顺序，像 GitLab Handbook 那样先解释背景、取舍和当前做法，再在必要处给出明确要求。

写作和审阅时，应检查页面是否回答了以下问题。简单页面可以用几个段落自然覆盖这些内容；复杂页面也可以把其中几项作为小节标题。

### 为什么值得写

页面是否说明了它要解决的协作问题、常见失败模式，以及不写清楚会造成什么成本。不要从工具功能或理想口号开始。

### 当前做法是什么

页面是否说明了团队当前如何处理这个问题。涉及明确义务时，应写清楚谁在什么条件下必须、应当或可以做什么。每条要求只表达一个主要义务，并尽量使用可观察的行为，例如“任务必须包含负责人”，而不是“任务应当清晰”。

### 边界在哪里

页面是否说明了紧急事件、敏感信息、低风险小改动等例外。如果无法穷举，应说明由谁判断以及判断后需要留下什么记录。

### 示例是否足够

每个会影响日常执行的页面都应尽量提供示例。容易误解的规则应同时提供反例，并指出问题在哪里。示例使用虚构或脱敏信息，不得泄露真实凭据、客户数据或个人信息。

示例不只是补充说明。它帮助人类理解边界，也帮助 Agent 校准判断尺度。若某页暂时没有示例，应在后续迭代中优先补上。

### 平台落地是否被隔离

页面是否先写平台无关的协作要求，再描述飞书、GitHub、GitLab 等平台如何满足它。平台落地说明可以放在同一页的短小段落中，也可以在内容较多时拆到独立页面；无论采用哪种方式，都应与规范正文清楚分开，并链接回对应正文。

不是每个页面都需要平台落地说明。只有当平台能力会影响实际执行，或读者需要知道如何把手册要求映射到具体工具时，才补充平台落地。平台落地不得扩展规范正文中不存在的强制要求。

### Agent 能否正确使用

页面是否说明 Agent 检查需要哪些输入、可以检查什么、不能可靠判断什么，以及发现问题后应如何建议。具体分级见“面向 Agent 的写法”。这些内容可以集中成小节，也可以放在相关规则附近。

## 写作要求

### 一页解决一个主要问题

标题应能让读者预测页面回答什么问题。若页面同时解释消息、任务、评审和决策，应先判断它们是否共享同一条上位原则；否则拆分并相互链接。

### 始终是草稿，描述当前做法

Handbook 没有最终版本，但合并后的正文应描述团队现在怎样工作。改变流程时直接修改 Handbook；有限范围的实验也可以写入正文，并在相应规则附近说明实验边界，结束后再删除说明。不要让“未来我们将……”被误解为当前做法。

### 给足上下文，保持段落短小

异步读者不能依赖作者在线补充背景。首次出现的术语应解释，代词应有明确指向，例外应靠近对应规则。段落宜短，但不要为了短而把完整论证切成口号或项目符号堆积。

### 使用具体、可观察的表述

推荐：

> 需要他人采取行动的消息必须明确写出行动、负责人和期望时间。

不推荐：

> 消息应该高效、清晰、及时。

前者可以执行和检查；后者缺少判断标准。

### 解释原因，但不重复规则

原因帮助团队在新场景中正确迁移规范。原因只需解释风险与取舍，不应把同一义务换一种说法重复多次。

### 保持术语一致

中英文术语首次出现时可以并列，例如“单一事实源（Single Source of Truth, SSoT）”。此后使用一个固定称呼。平台专有名词如 Issue、Pull Request 和 Merge Request 可以保留英文，不应为了形式统一制造难懂译名。

### 链接到权威来源

内部内容优先使用相对链接，链接文字应说明目标内容。不要使用“点击这里”作为链接文字。同一规范只能有一个权威位置；其他页面应链接引用，不得复制一份再分别维护。

外部资料用于提供证据和背景，不能代替本项目自己的规范。关键结论应在正文中完整表达，避免读者必须打开外链才能执行。

### 为目录提供入口

每个长期保留的目录都应有 `README.md`。目录 README 用于说明该目录的用途、内容边界和推荐阅读路径，帮助人类和 Agent 判断应该从哪里开始读。

目录 README 不应重复目录下正文的规则内容，也不应成为新的规范来源。若 README 中出现长期有效的协作要求，应将其移入合适的 Handbook 正文页面，并在 README 中链接过去。

### 不把关键含义藏在非文本内容中

图表、视频、截图和平台内嵌对象可以辅助解释，但关键规则、例外和结论必须同时用正文表达。图片应有说明性替代文本；视频应提供摘要或文字记录。人类或 AI 无法访问附件时，仍应能够理解规范的主要含义。

### 为规则提供可引用位置

一条规则应位于含义明确的标题下。需要被 Skill 或 Agent 引用的原子规则，可以获得固定标识；修改措辞时保留标识，改变含义时同步检查相关引用。标识用于引用，不应把正文写成难读的配置文件。

### 控制规则数量

每增加一条强制规则，都会增加执行和审阅成本。若一条建议不会显著降低协作风险，也无法帮助团队作出一致判断，就不应提升为“必须”。

## 规范正文与平台落地

规范正文定义协作要求和预期结果，平台落地定义实现路径。两者都可以写在 Handbook 中，但必须保持隔离：规范正文是要求来源，平台落地只是说明某个平台如何满足这些要求。例如：

| 层次               | 示例                                             |
| ------------------ | ------------------------------------------------ |
| 规范正文           | 影响后续行动的会议结论必须进入长期、可链接的载体 |
| 飞书落地           | 将结论写入飞书文档或任务，并在群聊中发送链接     |
| GitHub/GitLab 落地 | 将结论写入 Issue、PR/MR 或决策记录，并交叉链接   |

判断一句话属于哪一层，可以尝试替换工具名称：替换后含义仍然成立，通常属于规范正文；只有在某个平台中才成立，通常属于 Implementation。

平台落地与规范正文冲突时，应先判断是落地说明有误，还是正文忽略了平台限制。不得通过悄悄修改某个平台的示例来掩盖冲突；需要改变正文含义时，应按重大变更处理。

不要为了结构完整而给每个规范页面补齐所有平台。短小的平台说明可以放在规范页附近；当平台内容涉及多个对象、步骤较长或需要持续维护时，应拆到独立的落地页面，并从规范页链接过去。

## Handbook 与 Skill

两者解决不同问题：

| 组成     | 主要读者  | 职责                                                         |
| -------- | --------- | ------------------------------------------------------------ |
| Handbook | 人类与 AI | 解释为什么需要规范、规范要求、适用边界、例外、示例和平台落地 |
| Skill    | AI Agent  | 描述如何定位相关章节、补齐上下文、理解规范并准确引用         |

Handbook 是规范含义的唯一来源。Skill 不保存第二份规则，不把自然语言规范改写成另一套隐藏约束，也不规定具体检查算法。Handbook 发生结构变化时，只更新 Skill 中受影响的查阅路径。

Skill 应描述查阅手册的方法，而不是复制手册内容或替代 Agent 能力。一个 Handbook Skill 至少需要说明：

1. 如何根据协作产物和问题定位相关章节。
2. 如何沿内部链接查找上位原则、例外和平台落地说明。
3. 如何理解“必须”“应当”“可以”“不得”等规范用语。
4. 如何引用具体标题、规则标识或 Git 版本。
5. 找不到规则、规则冲突或上下文不足时如何向用户说明。

完成查阅后，Agent 根据当前任务、可用工具和自身能力决定如何分析或执行。Skill 不需要为消息、PR、任务等对象分别实现检查器。

## 面向 Agent 的写法

每项准备自动检查的规则，应标记其可检查性：

| 级别       | 含义                               | 示例                                   |
| ---------- | ---------------------------------- | -------------------------------------- |
| 确定性检查 | 仅凭结构或明确字段即可判断         | PR 是否关联 Issue，任务是否有负责人    |
| 启发式检查 | Agent 可以发现风险，但不能确定违规 | 请求是否提供了足够上下文，标题是否准确 |
| 人工判断   | 依赖业务权衡、权限或隐含背景       | 决策是否正确，截止时间是否合理         |

对于不能自动检查的规范，必须说明原因。AI 仍然可以阅读和解释这些规范，但不得为了让规则“可自动化”而把主观判断伪装成关键词或字数检查。

Agent 的审阅结果应包含：

1. 所依据的规范及其位置。
2. 在被检查内容中观察到的证据。
3. 问题或风险，而不是对人的评价。
4. 可执行的改进建议。
5. 对启发式判断给出不确定性，并允许人类忽略或纠正。

Agent 不得在没有规范依据时新增要求，不得代替负责人批准、拒绝或作出业务决策，也不得因无法访问受限上下文就断言内容不存在。

## 变更流程

### 小型变更

错别字、失效链接、语义不变的澄清和示例修正，可以直接提交 Pull/Merge Request。变更说明应简要写明原因。

### 较大变更

新增或改变义务时，默认仍然直接提交 Pull/Merge Request，让讨论围绕具体文本和 diff 发生。出现以下任一情况时，可以先使用 Issue：

- 目前只有问题和目标，还没有可以评审的具体文本。
- 需要先收集事实、案例或受影响团队的输入。
- 影响多个章节、策略或平台落地说明。
- 存在明显争议、迁移成本或安全与隐私影响。

Issue 适合尚无具体文本、需要先收集事实，或影响范围大到无法通过小型 MR 表达的情况。Issue 至少应回答：当前问题是什么、为什么值得规范化、可能的方案是什么、有哪些替代方案、谁会受到影响、如何迁移、Agent 能否检查，以及还有哪些未决问题。

### 讨论与合并

讨论可以发生在 Issue、PR/MR、飞书或会议中，但决定和关键理由必须回写到 Issue、PR/MR 或权威文档。聊天记录和会议录像可以作为补充证据，不应成为理解变更所必需的唯一来源。

审阅者应分别检查：

- 这项规则是否解决真实且重复的问题。
- 表述是否清楚、可执行，并包含必要例外。
- 是否与现有规范重复或冲突。
- 规范正文与平台落地是否被正确分开，平台落地是否没有新增正文义务。
- Agent 可检查性是否被诚实描述。
- 示例和链接是否安全、有效且易于理解。

合并表示该变更成为当前规范。重要变更应在团队使用的沟通平台中发布链接和简短影响说明，但公告本身不替代权威文档。

## 常见反模式

- **把聊天当档案**：关键决定只存在于群聊或私聊中。
- **复制而不引用**：多个页面分别维护同一项规则。
- **工具冒充规范**：把飞书或 GitLab 的按钮操作写成所有团队都必须遵守的原则。
- **愿景冒充现状**：尚未采用的流程被写成团队当前已经执行的做法。
- **字段替代思考**：模板字段很多，但没有解释为什么需要这些信息。
- **自动化过度承诺**：把需要业务判断的问题标记为确定性检查。
- **只写正确示例**：读者无法识别常见但有问题的边界情况。

调查得到的核心判断如下：

### 直接借鉴

- 整本 Handbook 始终是草稿，但合并后的内容表达当前做法；记录当前做法使改变更容易，而不是使流程僵化。
- Handbook 应是当前流程的单一事实源，而不是散落信息的索引或愿景文章集合。
- 改变流程时同步修改 Handbook，并通过版本控制和 MR 保留讨论与变更历史。
- 会话和会议中的结论应写回长期载体，传播时链接到变更或权威页面。
- 同一内容避免重复维护；任何参与者都可以通过 MR 修正内容，使它持续反映当前做法。
- 自动构建、链接检查、Markdown lint 和敏感信息检测适合作为后续质量保障。
- 反馈应针对工作和行为而非人格，分歧应假设善意，复盘应关注机制而非归咎个人。
- 异步流程应照顾时区、语言、认知方式和可持续工作，效率不能以长期在线或加班为代价。

### 调整后借鉴

GitLab 倾向“公开优先”。本项目改为“在授权范围内默认可发现”，因为中国 IT 团队常处理客户数据、个人信息、内网系统和商业敏感内容。透明度应服务协作，不应越过必要的权限与合规边界。

GitLab 强调从 Merge Request 开始。本项目保留“可审阅、可追踪的变更”这一原则，但不把 GitLab MR 设为唯一入口。使用 GitHub、Gitea、飞书或其他平台的团队可以采用等价流程。

GitLab 的 Handbook 同时承载大量公司运营内容。本项目只聚焦协作手册内容，避免扩展到人事、法务、财务和组织宣传。

GitLab Values 使用 CREDIT 品牌框架并混合公司经营目标。本项目只提取能够约束协作行为的人文原则，不复制品牌、价值排序或员工评价体系；这些原则也不得被 Agent 转换成人格评分。

### 不直接借鉴

- GitLab 的组织目录、岗位审批和内部维护流程。
- Slack、GitLab Pages、Hugo、Docsy 等具体工具选择。
- “任何信息默认对公网公开”的组织级承诺。
- 仅因 GitLab 公司文化或治理结构才成立的规则。

GitLab 的经验是重要样本，不是本项目规范的权威来源。每项被采用的规则仍必须说明它解决了本项目目标团队的什么问题。完整逐页索引和摘要见[GitLab Handbook 相关内容导览](research/gitlab-handbook-outline.md)。
